# TypeScript → Rust port plan for midnight-did

Survey of the `midnight-did` TS develop branch (commit `e5bcf02`). Maps every
TS source/test file in `contract/`, `domain/`, `did/`, and `api/` to a Rust
target, recommends a crate layout under `midnight-did-rs/crates/`, and proposes
a phased port plan that lands a bedrock domain layer first and saves the
wallet/provider plumbing for last.

## TL;DR

The four TS packages we target weigh in at **5,482 LOC of source** plus
**7,338 LOC of tests** (28 test files). The contract package is small
(0.5k LOC of `.compact` + 0.1k LOC of TS witnesses; the rest is generated zkir
and the simulator that we will not port). The domain layer is the biggest pure
chunk (~1.9k LOC) and consists almost entirely of zod-driven schema validation
plus a non-trivial offchain DID document encoder (`offchain-midnight.ts`,
722 LOC) that round-trips an `OffchainMidnightDIDState` through a `MOD1`-tagged
binary frame to a base64url long-form DID. The `did/` package (~900 LOC) layers
Midnight-specific validation on top of the generic DID Core types and supplies
`LedgerToDomain` — the canonical contract-ledger → DID-document mapping
that anything we generate from `did.compact` will be measured against. The
`api/` package (~2.6k LOC) is mixed: about a third is wallet/dust/SDK plumbing
that should NOT port directly (it is bound to the wallet-sdk-facade and ledger-v8
TS SDKs), about half is operation builders + ledger mappers + private-state
that DOES port, and the rest is config/logger glue.

Recommended structure: **four crates**, all under `crates/`:
`midnight-did-domain` (pure types + schema validation, no I/O,
no Compact runtime), `midnight-did` (the codegen target — keep the existing
crate name, wraps `generated.rs` and re-exports `Ledger`/`Contract`/circuit
APIs plus the hand-written `controller-key`/`witnesses` shim that mirrors the
TS source), `midnight-did-method` (`LedgerToDomain` + Midnight DID Document
+ resolver + offchain decoder — TS `did/` package), and `midnight-did-api`
(operation builders, ledger-state reads, private-state, network mapping,
optional wallet/provider integration behind a feature flag). Total port effort
**~12–18 dev-days**: 1d for crate skeletons + Cargo wiring, 3–4d for domain
(schema-equivalent validation is fiddly), 2d for the codegen handoff
(currently DID-1/2 — assumed in flight), 3d for `midnight-did-method`
(LedgerToDomain mappers + Midnight document validation), 3–4d for the api
operation layer + ledger-state reads + ledger-mappers, 2d for the test port
(prioritise `did.test.ts`, `ledger-to-domain.unit.test.ts`,
`ledger-mappers.test.ts`, `verification-method-relations.test.ts`,
`controller-operations.test.ts`, `private-state.test.ts`). Biggest risks: (1)
zod-style validation is a porcelain feature in TS — porting all the consistency
checks in `validateDIDDocumentConsistency` faithfully is detail-heavy; (2) the
offchain MOD1 framing in `offchain-midnight.ts` is hand-rolled binary that
must be byte-identical for long-form DIDs to round-trip across runtimes; (3)
wallet/provider integration in `api/` couples to multiple wallet-sdk-* TS
packages with no public Rust equivalent yet (these can be deferred to a
follow-on crate or stubbed behind a trait).

```

├── midnight-did-domain/        # pure types, schema validation, no I/O
│   ├── src/
│   │   ├── lib.rs
│   │   ├── did_document.rs     # ports did-document.ts (~700 LOC TS)
│   │   ├── crypto_codecs.rs    # base64/base64url + FieldCodec
│   │   ├── midnight.rs         # MidnightNetwork + did:midnight: schema
```

Rationale for the boundaries:

- **`midnight-did-domain`** has zero Compact-runtime or wallet dependencies. It
  is a candidate crate to publish independently and is the layer that DID
  consumers (resolvers, VC libraries) will depend on. Schema validation lives
  here.
- **`midnight-did`** stays as the codegen target (current crate). It owns
  `controller-key.rs` and `witnesses.rs` because both shadow `.compact`
  declarations and need the same `compact-runtime` deps. Splitting the
  generated code into its own crate would bring the codegen surface area into
  every consumer; better to re-export from `midnight-did`.
- **`midnight-did-method`** sits between `midnight-did-domain` and
  `midnight-did` and supplies the Midnight method profile (`did:midnight:*`
  constraints) + the canonical `LedgerToDomain` mapper. Splitting this from
  the API keeps the resolver path free of any wallet/provider dependencies.
- **`midnight-did-api`** is the operation/orchestration layer. The
  wallet/provider code stays here (feature-gated) so the rest of the stack
  builds without the wallet-sdk dependency.

Alternative: collapse `midnight-did-method` into `midnight-did-domain` if the
upstream prefers fewer crates. We recommend keeping it separate because the
Midnight method validation depends on the generated contract types
(`DIDContract.CurveType` etc.) — pulling those into `domain` would block its
no-dep publishing.

## Port plan in phases

Each phase is sized for 1–3 dev-days. Phases 1, 2a/2b, and 3 are independent
once the codegen handoff lands.

### Phase 1 — Crate skeletons + domain bedrock (2 days)

- Create `midnight-did-domain` with `crypto_codecs`, `midnight`,
  `uri_normalization`, `ledger_utils`, and the minimal `did_document` subset
  needed by `LedgerToDomain` (parse/createDID, parseDIDURL, parseDIDKeyID,
  PublicKeyJwk + VerificationMethod + Service types — leave the
  cross-consistency validator for Phase 1b).
- Port `crypto-codecs.test.ts`, `ledger-utils.test.ts`,
  `uri-normalization.test.ts`, `midnight.test.ts`.
- Add `midnight-did-method`, `midnight-did-api` skeletons (empty libs, just to
  validate the workspace builds).

### Phase 1b — Full domain validation (2 days)

- Port `validateDIDDocumentConsistency` (uniqueness, relation-reference,
  service-id, endpoint-uniqueness checks).
- Port the offchain framing (`encodeOffchainMidnightDIDState`,
  `decodeOffchainMidnightDIDState`, long-form parsing).
- Port `did-document.parse.test.ts`,
  `did-document.verification-method.test.ts`,
  `did-document.document-resolution.test.ts`, `offchain-midnight.test.ts`.

### Phase 2 — Contract shim (1–2 days, depends on DID-1/DID-2)

- DID-1/DID-2 generate `crates/midnight-did/src/contract/generated.rs` from
  `did.compact`.
- Hand-port `controller-key.ts` to `controller_key.rs` (32-byte pad + persistent
  hash). Capture a TS reference vector and add a byte-parity test.
- Hand-port `witnesses.ts`: `DIDPrivateState { secret_key: [u8;32] }`,
  implement `WitnessImpl` so the generated contract can call `local_secret_key`
  and `current_timestamp` (plus `get_schnorr_reduction` if Schnorr-circuit
  parity is in scope for v1).

### Phase 3 — LedgerToDomain + Midnight document (3 days)

- Port `ledger-to-domain.ts` (the 497-LOC mapper) into
  `midnight-did-method/ledger_to_domain.rs`. Use the generated `Ledger` reader
  methods (Counter view, Set member-iter, Map entries) wired through whatever
  `Ledger`-view interface `midnight-did/src/contract/generated.rs` exposes.
- Port `midnight-did-document.ts` (`MidnightDIDDocumentSchema` rules: required
  contexts, single-controller, curve restrictions).
- Port `ledger-to-domain.unit.test.ts` + `midnight-did-document.test.ts` +
  `midnight-did-resolver.test.ts` (the resolver takes a mocked
  `LedgerReader` trait).

### Phase 4 — API operations (3–4 days)

- Port `types.rs` (provider trait shape — pick `MidnightDIDProviders` as a
  trait family `Providers`, with associated `LedgerReader`, `PrivateStateStore`,
  `CallTx` types).
- Port `network-mapping`, `did-subject`, `ledger-mappers`,
  `verification-method-relations`, `ledger-state`, `resolution`,
  `private-state`, `controller-operations`, `verification-method-operations`,
  `service-operations`, `document-operations`.
- Port `commons.ts` fixtures (subset needed by tests below) +
  `ledger-mappers.test.ts`, `verification-method-relations.test.ts`,
  `controller-operations.test.ts`, `private-state.test.ts`,
  `verification-method-operations.test.ts`,
  `runtime-to-domain.test.ts`, `domain-to-runtime.test.ts`, `types.test.ts`.

### Phase 5 — Acceptance suite + byte-parity (2 days)

- Port the 27 scenarios in `did.api.test.ts` to Rust integration tests
  against a Rust simulator harness that drives the generated contract
  in-process (no wallet/proof-server). This is the headline CRUD-coverage
  suite.
- Capture TS `ContractState` byte snapshots from `contract/src/test/did.test.ts`
  at six points (see "Byte-parity opportunities" below) and assert the Rust
  ledger emits identical bytes after the same operation sequence.

### Phase 6 (deferred) — Wallet/provider integration

- Define `Providers` traits in `midnight-did-api` (zk-proof, public-data,
  private-state, wallet) so a downstream crate can wire them to Rust
  equivalents of the TS wallet-sdk-* packages when those become available.
- Provide a stub feature `wallet-mock` for tests; do not port the TS
  wallet-sdk-facade code until upstream publishes a Rust wallet SDK.

## Byte-parity opportunities

The Rust port has byte-parity leverage anywhere the TS test suite hits a ledger
snapshot. We capture TS `ContractState` (or just its
`storage`-equivalent serialisation) at each labelled point and assert the Rust
side after running the same sequence. Concrete fixtures to capture from the
existing TS suite:

| Fixture name | After operation | Source test | Byte target |
| --- | --- | --- | --- |
| `did_initial` | Constructor with fixed secret key + timestamp witness. | `did.test.ts` "properly initializes ledger state". | `controllerPublicKey`, `created`, `updated`, `active`, `contractVersion=1`. |
| `did_after_rotate` | `rotateControllerKey(new_pk)`. | "rotates the controller key". | `controllerPublicKey`, `version+=1`, `operationCount+=1`, `updated`. |
| `did_after_add_vm` | `setVerificationMethod(vm, Insert)`. | "should add a verification method". | `verificationMethods` map entry. |
| `did_after_add_schnorr_vm` | `setSchnorrJubjubVerificationMethod(...)`. | "should add ... SchnorrJubjub". | `schnorrJubjubVerificationMethods` map entry. |
| `did_after_add_relation` | `setVerificationMethodRelation(Authentication, methodId, Insert)`. | "should add Authentication relation". | `authenticationRelation` set. |
| `did_after_add_service` | `setService(svc, Insert)`. | "should add a service". | `services` map entry. |
| `did_after_add_aka` | `setAlsoKnownAs("urn:example:1", Insert)`. | "should add an alsoKnownAs". | `alsoKnownAs` set. |
| `did_after_deactivate` | `deactivate()`. | "should deactivate the DID". | `active=false`, `deactivated=true`. |
| `did_multi_op` | Insert VM → Add 5 relations → Add service → AKA → rotate. | "Multiple Operations" / "version counter". | Counters monotonically advance; controllerPublicKey changes. |

Capture mechanism: extend `contract/src/test/did-simulator.ts` to dump
`Buffer.from(contractState.toString())` (or the binary serialised form) at
each step, write into a shared `tests/fixtures/golden/` directory under
`midnight-did-rs`. The Rust integration test reads the same fixture and
diffs.

Additional byte-parity points outside the ledger:

- `deriveControllerPublicKey([0u8;32])` and a handful of other fixed secret
  keys — direct vector. Mirrors the in-circuit `controllerKey()` digest.
- `encodeOffchainMidnightDIDState(state)` → bytes — golden vectors from
  `offchain-midnight.test.ts`.
- `ledgerStateToDIDDocument(ledger, network, contractAddress)` → canonical
  JSON. Compare TS `JSON.stringify` (with BigInt replacer) to Rust
  `serde_json::to_string` for matching field ordering (use
  `serde_json::Value`-based assert to avoid ordering pitfalls).

## Risks + open questions

- **Schema validation faithfulness.** zod schemas in
  `did-document.ts`/`midnight-did-document.ts` chain many `.check(z.refine(...))`
  rules. Each rule must be re-implemented faithfully in Rust including the
  error-message text if we want the test suite to port verbatim. Recommended
  interpretation: keep the same error categories, drop verbatim message text;
  port the assertion semantics and adjust test expectations to use
  `error.kind()` rather than exact strings.
- **Offchain MOD1 framing.** `offchain-midnight.ts` rolls its own binary
  format (`MOD1` magic, length-prefixed chunks via `CompactType*`). Risk: any
  alignment-aware encoding divergence (cf. Prod-15 `Bytes<N>`-from-pad
  divergence already fixed in codegen) reappears here. Mitigation: golden
  binary vectors captured from TS, checked in `midnight-did-domain`.
- **Wallet/provider plumbing.** Currently no Rust equivalents for
  wallet-sdk-facade, hd, shielded/unshielded/dust wallets, or the
  level-private-state provider. We do NOT need them to validate
  `LedgerToDomain` + circuit calls, but we DO need them for `did.api.test.ts`
  to run end-to-end. Recommended interpretation: Phase 5 runs an in-process
  simulator (the `compact-runtime` Rust simulator already used for the
  zerocash/election fixtures), bypassing the wallet/proof-server entirely; the
  wallet integration is a separate downstream effort.
- **`LedgerToDomain`'s reliance on the generated `DIDContract.*` namespace.**
  The TS code reads from `DIDContract.CurveType` etc. — the Rust codegen must
  emit equivalent enums + accessor methods on `Ledger`. DID-1 (in flight) will
  enumerate the gaps; this plan assumes that survey will close them.
- **Schnorr witness `getSchnorrReduction`.** The TS witness signature accepts a
  `bigint challengeHash` and returns `[bigint, bigint]`. The Rust port needs
  to choose a representation (`U256`? `BigUint`? two `u128`?). This depends on
  what the generated contract expects from the witness — answer when DID-1
  finishes.
- **Single vs multi-controller.** TS Midnight method enforces single-controller
  equal to the DID subject. The on-chain `controllerPublicKey` is a single 32-
  byte field, so this is a domain-validation rule rather than a contract rule;
  the Rust port should keep it in `midnight-did-method`.
- **Service endpoint canonicalisation order.** TS uses
  `JSON.stringify(normalizeServiceEndpoint(endpoint))`. Rust must match the
  same field ordering (object-property insertion order). Recommended
  interpretation: implement the canonicalisation as a custom serde serializer
  that mirrors V8 insertion-order semantics OR pre-sort recursively. Confirm
  with a golden vector from `domain/test/uri-normalization.test.ts`.
- **`alsoKnownAs` URI rules.** TS requires absolute URIs; Rust's `url` crate
  is the natural choice but its definition of "absolute" differs slightly
  (no fragment, etc.). Recommended interpretation: encode the same checks
  with `url::Url::parse()` + post-conditions on scheme/host; flag any
  divergence as a test mismatch and decide.
