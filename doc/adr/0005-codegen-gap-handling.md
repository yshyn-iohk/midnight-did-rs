# ADR 0005 — Codegen-gap handling strategy

**Status:** Accepted
**Date:** 2026-06-03

## Context

The Rust port's on-chain contract crate is generated by
`compactc --rust` from `did.compact` (375 LOC of Compact source). The
codegen-rust toolchain has shipped enough body-emission walker arms to
cover the constructor + all 12 non-exported pure circuits in
`did.compact` (persistent-hash with mixed-pad+arg, Boolean ledger
reads in asserts, Map.member / Set.member in expressions, if-else-if
returning Map.member dispatched on enum, cross-circuit pure + impure
calls, nested-struct-field enum equality, the full set of disclose +
witness + cell-write shapes inside the constructor).

What does **not** emit cleanly is the body of 11 of the 23 exported
impure circuits. The
[`codegen-gap survey`](file:///Users/ysh/obsidian/midnight-mobile/Rust%20DID%20port/Research%20%E2%80%94%20codegen%20gaps%20for%20did.compact.md)
classifies these into five distinct IR shape closures:

- **Shape A — single ledger-cell write to non-Uint type in exported
  body.** `Bytes<N>` (in `rotateControllerKey`) and `Boolean` (in
  `deactivate`). 2 of 11.
- **Shape B — ADT mutation on `Opaque<"string">` element/key with a
  disclose-to-const prologue.** `Set.insert`, `Map.remove`, `Map.lookup`
  on `Opaque<"string">`. Affects `setAlsoKnownAs`,
  `removeVerificationMethod`, `removeSchnorrJubjubVerificationMethod`,
  `removeService`, and partially `verifySchnorrJubjubDigestSignature`.
  4-5 of 11.
- **Shape C — if-else-if Map mutation followed by unconditional
  Map.insert** with struct-field-derived key. Affects
  `setVerificationMethod`, `setSchnorrJubjubVerificationMethod`,
  `setService`. 3 of 11.
- **Shape D — prefixed-import circuit call from body + Map.lookup-to-const
  + Boolean ledger read in assert.** `verifySchnorrJubjubDigestSignature`.
  1 of 11.
- **Shape E — enum-inequality assert + Boolean-result-from-cross-pure-call
  bound to const.** `setVerificationMethodRelation`. 1 of 11.

The walker arms required to close A, B, and C are tractable
extensions of existing walker patterns (A mirrors the `tiny.compact`
5-stmt body arm but with a 1-stmt RHS; B extends the F-set fixture's
`Set<Bytes<32>>.insert` arm to `Opaque<"string">`; C combines E4-class
ADT-mutation arms with E6.2's if-else-if-mid-body work). D and E need
dedicated fixtures (Schnorr prefixed-import; cross-pure-call to const).

Three options exist for the DID work:

1. Block all DID work on codegen-rust closing every gap.
2. Hand-write the entire contract module in Rust, ignoring the
   codegen.
3. Stub the gap-affected circuit bodies in a working-copy `.compact`
   so codegen emits a buildable `generated.rs`, and ship the rest of
   the stack (api, domain, tests). Hand-fill the stub bodies in a
   sibling module as needed, while the codegen-rust gaps are closed
   in priority order.

Option 1 is multi-week-blocking calendar work. Option 2 discards the
regeneration story — every future `did.compact` change requires
hand-merging into the Rust transcription.

## Decision

Use **option 3**. Concretely:

- A **working copy** of `did.compact` lives at
  `target-gen/did-gap-survey-tree/packages/contract/src/did.compact`.
  The 11 affected exported circuit bodies are replaced with a minimal
  buildable stub (`contractVersion = 1;` or equivalent).
- `compactc --rust` runs against the working copy to emit
  `crates/midnight-did/src/contract/generated.rs` (1,320 LOC). The
  resulting module compiles standalone — every type, witness, ledger
  view, the constructor, and the 12 non-exported circuits emit
  cleanly.
- Hand-written replacements for the 11 stubbed bodies will live in a
  sibling module
  [`crates/midnight-did/src/contract/extensions.rs`](../../crates/midnight-did/src/contract/)
  (to be added). Each shim uses `compact-runtime`'s `OpProgramVerify`
  builders to assemble the same transcript the generated code would
  produce. The api crate's `DidContract` production impl will pick
  the shim over the stubbed `generated.rs` body for the 11 affected
  circuits.
- The full codegen-gap report (with shape classification, individual
  gap details, and closure effort estimates) lives in Obsidian at
  `Research — codegen gaps for did.compact.md`. It tracks one row per
  gap (11 rows) and one per shape group (5 rows) with concrete
  walker-arm recipes.
- As codegen-rust closes gap shapes (in priority order: A → B → C →
  D → E), the corresponding hand-shimmed circuits are replaced by
  the regenerated emission. The shim file shrinks toward empty over
  time.

## Alternatives considered

**Patch codegen-rust to close every gap before any DID work.**
Rejected. The five shape closures collectively are multi-week effort
(A is S/M, B is M, C is L, D is L, E is L); blocking DID delivery on
all five means the api crate cannot ship, the domain crate has no
upstream consumer yet, and zero test coverage exists for the DID
operation surface. Splitting the timelines lets us ship the rest of
the stack now and close codegen gaps incrementally without
calendar-pressure.

**Hand-write the entire contract module.** Reject the codegen
entirely; transcribe `did.compact` to Rust by hand using
`compact-runtime` builders. Rejected. Drift cost is high — any
`did.compact` change requires a full re-port, with no mechanical
guarantee of byte-parity. Loses the regeneration story that the
codegen-rust toolchain was built to deliver. Also: for the 12
non-exported circuits that emit cleanly, this would mean throwing
away 700+ LOC of already-correct, machine-generated code.

**Hand-write all 11 shims now, then regenerate.** Skip the
working-copy `.compact` stubs; immediately write the 11 hand-shimmed
bodies. Rejected as a first step. Many of the 11 share IR shape
closures with each other (B-shape covers 4-5 circuits; C-shape
covers 3); hand-writing all of them is duplicative. Better to wait
until the priority order is set (likely "what does the wallet need
first?") and shim only what is on the critical path.

**Use the TS simulator (via wasm-bindgen) for the 11 affected
operations.** Rejected outright. The TS reference uses a TS-only
in-memory simulator (`packages/contract/src/test/did-simulator.ts`)
that does not run real Compact circuits. Wiring that into Rust would
build a hybrid runtime with no path to byte-parity against the
production contract.

**Stub the gap-affected circuits to `unimplemented!()` instead of a
minimal valid body.** Rejected. `unimplemented!()` in the emitter
output is a hard compactc error (per Prod-3 in the codegen-rust
toolchain). The current stub (`contractVersion = 1;`) emits valid
Rust that panics at runtime when actually called — discoverable.

## Consequences

**Positive:**
- The api crate ships with 92 tests passing today, exercising the
  full operation surface via `RecordingContract`. The codegen-gap
  status does not block api-level delivery.
- `generated.rs` compiles standalone — `cargo build -p midnight-did`
  fails only on the upstream halo2 issue (out of scope for this
  ADR), not on the codegen gaps themselves.
- Codegen-gap closure happens in shape-priority order (A → B → C →
  D → E) with each closure unblocking multiple circuits at once.
- Regeneration story is preserved. Every `did.compact` change reruns
  `compactc --rust`; the only working-copy delta is the stub bodies,
  applied as a small patch.

**Negative:**
- The 11 stubbed circuits are non-functional at the production
  `DidContract` impl level until either (a) the shim file is filled
  or (b) codegen closes the corresponding shape. The api crate is
  unaffected (it uses `RecordingContract`).
- Two sources of truth for the contract body: `did.compact`
  (canonical) and `extensions.rs` (hand-shimmed). Drift risk if
  `did.compact` changes a circuit signature and the shim is not
  updated. Mitigation: a doc comment on each shim points to the
  `did.compact` line range, and a follow-up CI check (TBD) can
  compare signatures.
- The codegen-gap survey is a living document that has to track each
  closure. Mitigation: the survey is in Obsidian (not the repo) where
  it can be updated cheaply; each closure references the relevant
  walker-arm recipe.

**Locked in:** The strategy (stub-then-shim-then-regenerate). The
specific shim implementations are not locked in — they are by
construction replaceable by regenerated emission.

**Preserved flexibility:** Order of gap closure; whether to fully
shim a given circuit in `extensions.rs` or wait for codegen.
Per-circuit decision made by whoever picks up the next iteration.
